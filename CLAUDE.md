# Правила работы над этим проектом

Полная версия целей и контекста — в `project.md`. Здесь — обязательные к
соблюдению правила.

## Юридические рамки

- Весь код, графика и звуки — оригинальные. Никаких чужих ассетов и спрайтов.
- Названия игр — собственные, не использующие чужие торговые марки
  (не «Tetris», не «Pac-Man», не «Pong» и т.п.).
- Игру типа «падающие фигуры» в классическом виде не делаем (поле 10×20 и
  вид фигур — trade dress The Tetris Company). Только с существенно
  изменённой механикой и визуалом.

## Порядок работы

- Одна игра — end-to-end (игра → отправка счёта → серверная валидация →
  таблица лидеров → запуск), прежде чем начинать следующую.
- Общий код выносится в `core/` только в момент второго использования, не раньше.
- Маленькие шаги: один шаг — одна задача, после каждого шага — короткое
  объяснение и подтверждение перед следующим.

## Архитектура

- Монолит: один сервер, одна БД (SQLite / `node:sqlite`).
- Структура по фичам (`games/<имя>/`, `server/`), не по типам файлов.
- Импорты — только от игр к `core/`, не наоборот. Без циклических импортов.
- У каждого куска состояния — один модуль-владелец.
- Файлы 200–400 строк — норма, не дробить преждевременно.

## Безопасность (обязательно)

- Никогда не доверять счёту от клиента: серверный токен сессии + проверка
  максимально возможного прироста очков за время сессии.
- Rate limiting на отправку счёта и старт сессии.
- Ники и любой пользовательский ввод — только через `textContent`/
  `createElement`, никогда `innerHTML`. Валидация ника по белому списку
  символов на сервере.
- Только параметризованные SQL-запросы.
- Секреты — в `.env`, `.env` в `.gitignore` с первого коммита.
- Топ лидерборда ограничен (top-100).

## Стиль ответов

- Код — по одному файлу за раз, с полным путём.
- Перед крупным решением — 2–3 предложения «почему так» и одна альтернатива.
- Если запрос нарушает эти правила — остановиться и объяснить риск, не
  выполнять молча.
